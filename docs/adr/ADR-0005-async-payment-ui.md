# ADR-0005: 決済結果を非同期確定とし、UIはポーリングせず支援履歴で確認する

## 状態

採用

## 背景

決済は非同期モデルである。支援申込は `202 Accepted` を返し（`paymentStatus` は初期 `PENDING`/`PROCESSING`）、
最終確定は決済プロバイダの Webhook（→ Outbox/バッチ経由の状態遷移、基本設計 §6.8/§8）で行われる。
支援結果画面（SCR-042）で「即時に確定しない決済結果」をどう扱うかを決める必要がある。

選択肢:

1. 申込後、確定するまで画面で同期待ち（サーバー側で待機 or クライアントでポーリング）
2. **申込を受け付けたら結果画面で受付を通知し、最終状態は支援履歴（SCR-051）で確認させる（非ポーリング）**
3. WebSocket / SSE でサーバーからプッシュ通知する

## 判断

**選択肢2（非同期受付 + 支援履歴で確認、ポーリングなし）** を採用する。

## 判断理由

- バックエンドの非同期設計（Webhook → Transactional Outbox → バッチ）と整合する。UIが確定を同期待ちすると、
  リクエストの長時間保持やタイムアウト、プロバイダ遅延時の破綻を招く（選択肢1の問題）。
- ポーリングは実装・負荷ともに増える割に、教育用スコープでは体験上の利点が小さい。確定状態は
  支援履歴（SCR-051）と支援詳細で確実に参照できる。
- プッシュ（選択肢3）はリアルタイム性が高いが、SSE/WebSocket基盤・接続管理・認証の追加が必要で、
  現スコープに対して過剰。将来リアルタイム表示が要件化した時点で再検討する。

## 適用範囲

- SCR-042 結果画面: 受付（`202`）と現在の `paymentStatus` を表示し、「決済結果は非同期で確定します。
  最新状態は支援履歴で確認してください」と案内する（`SupportFlow` の result ステップ）。
- SCR-051 支援履歴 / 支援詳細: 確定後の状態を表示する。

## 結果

- `SupportFlow.tsx` の result ステップが受付通知＋支援履歴への導線を提供（ポーリング実装なし）。
- 決済確定の一連（Webhook→状態遷移→通知）はバックエンド結合テスト
  （`PaymentWebhookFlowIntegrationTest` 等）で網羅。
- リアルタイムプッシュ（SSE/WebSocket）は将来課題として保留。
